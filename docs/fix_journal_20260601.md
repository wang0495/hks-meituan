# 调优随笔：从65分到81分的漫长爬坡

> 2026-06-01 | CityFlow 路线规划系统优化记录

## 背景

CityFlow 是一个基于 MoE（Mixture of Experts）Agent 架构的智能城市出行规划系统。核心流程：

```
用户自然语言 → rule_guard(结构校验) → expert_router(场景分类)
→ Wave1/Wave2 专家团提案 → review ↔ rework → synthesizer(路线组装)
→ LLM时间修正 → 文案生成 → 输出
```

覆盖珠海 2000+ POI，5种场景类型：美食型、目的地型、特种兵型、休闲型、观光型。

跑 100 场景全量测试，均分 65.2，F 级 8 个。决定开调。

---

## 第一阶段：评分体系自身的 bug（假 0 分）

### 现象

报告中 5 个维度（地理连贯/多样性/完整性/时间可行/类别匹配）对美食型/目的地型/特种兵型几乎全 0。

### 排查

以为是路线质量差，反复改 synthesizer 逻辑。后来发现：

**LLM 评分 prompt 只有 4 个维度（完整性/场景匹配/逻辑合理/体验丰富），规则评分有 6 个维度（完整性/类别匹配/地理连贯/时间可行/多样性/POI质量）。** 报告合并时，LLM 没输出的 2 个维度直接填 0。

不是路线烂，是评分表印错了。

### 修复

- 统一 LLM 评分 prompt 到 6 个标准维度
- 加 `_normalize_dims()` 桥接函数，旧维度名自动映射：
  - "场景匹配" → "类别匹配"
  - "逻辑合理" → "时间可行"
  - "体验丰富" → "多样性"
- 缺失维度用已有维度均值填充，不再给 0

**教训**：当你看到某个维度全 0，先怀疑数据采集，再怀疑算法逻辑。

## 第二阶段：评分 LLM 在撒谎（地理连贯假扣分）

### 现象

美食型路线被 LLM 评为"地理跳跃严重，折返跑"，地理连贯 45 分。特种兵型也被评"跨区折返"，45 分。

### 排查

手算实际站间距离：

```
美食型路线:
  翠香路老茶餐厅 → 情侣路海滨长廊: 1.1km
  情侣路海滨长廊 → 香洲渔港: 0.0km (同一位置不同POI)
  香洲渔港 → 聚潮轩: 1.9km
  聚潮轩 → 德壹记凉茶: 0.1km
  德壹记凉茶 → 梅华路夜市: 0.7km
```

**最大站间距 1.9km，全部在香洲区 2km 范围内。** 根本没有跳跃。

再看 LLM 评分的输入 — `_format_route_text()` 只给了名称、类别、价格、时间、评分。**没给坐标。**

评分 LLM 全凭名字猜"在哪"，然后判定"梅华路到香洲很远" — 其实只隔 700 米。

### 修复

- `_format_route_text()` 加上坐标和站间距离：
  ```
  1. 翠香路老茶餐厅 [餐饮] 评分:4.2 坐标:(22.2710,113.5520) 到达:08:00 离开:09:10
  2. 情侣路海滨长廊 [景点] 评分:4.5 坐标:(22.2789,113.5456) 到达:09:25 离开:10:30 距上站:1.1km
  ```
- 评分 prompt 加硬规则：
  - 站间<3km: 满分(90-100)
  - 站间3-8km: 合理(70-89)
  - 站间>15km: 跨区(0-49)
  - "不要凭名字猜测距离！必须看坐标"

### 效果

| 场景 | 修复前地理连贯 | 修复后 |
|------|---------------|--------|
| 美食型 | 45 | **100** |
| 休闲型 | 70 | **88** |
| 观光型 | 60 | **95** |

**教训**：LLM 不知道的事情会编。给它事实（坐标），它就能判对。不给，它就猜，然后自信地猜错。

## 第三阶段：凉茶的冤案（子类误分类）

### 现象

美食型路线多样性差，"德壹记凉茶"被分到"茶餐厅/甜品"子类，导致看起来和茶餐厅重复。

### 原因

子类匹配逻辑是关键词包含：`"茶餐厅/甜品": ["茶餐厅", "甜品", "奶茶", ...]`。"凉茶"包含"茶"，命中了茶餐厅子类。

### 修复

- 加了凉茶专属关键词列表 `["凉茶", "草本", "龟苓膏", "苦茶"]`
- 子类匹配时先检查凉茶关键词，优先匹配"饮品/凉茶"
- 从茶餐厅/甜品子类中移除"饮品"关键词，避免交叉命中

**教训**：中文关键词匹配是坑。"凉茶"不含"茶餐厅"但包含"茶"。"东山湖"包含"东山"但不含"山东"。关键匹配要分层，先精确后模糊。

## 第四阶段：LLM 学不会拒绝（核心洞察）

### 现象

海岛跳岛、2小时极限短途、特种兵15站 — LLM 永远试图满足，从不说不。

- "珠海跳岛一日游3个岛" → LLM 硬凑出跨岛路线（实际渡轮来回6小时）
- "2小时吃遍珠海" → 塞8个POI
- "特种兵打卡15个景点" → 照着15站排

### 讨论过程

我最初想加一个 "feasibility gate" 架构组件，在 rule_guard 之后检测不可行请求。

用户一句话点醒：**"是不是你的 prompt 就没教会 LLM 拒绝？先教下试试，不行再说不行。"**

对。先试 prompt，不行再加架构。别过度设计。

### 修复

**路线编排 prompt** 加了最顶部的"拒绝"指令：

```
【最重要】你有权拒绝不合理的编排：
- 如果总时间<3小时，最多排2-3站，不要硬塞更多
- 如果景点分散在多个岛/区且交通时间>2小时，只保留一个区域
- 如果用户要求跳岛（如东澳岛+外伶仃岛），只选一个岛
- 如果候选POI地理跨度过大（>30km），只选最紧凑的一个区域
- 宁可排一条短而精的路线，也不要排一条长而散的烂路线
```

**时间修正 prompt** 加了"砍站"机制：

```
【最重要】如果站点太多塞不进时间范围，你有权砍站：
- 优先砍：距离远的、评分低的、与主题关系弱的
- 砍站后在dropped字段说明
- 宁可排一条时间宽裕的短路线，也不要拥挤路线
- 海岛类行程：渡轮单程至少60分钟，务必预留
```

LLM 输出格式也改了，支持 `dropped` 字段：

```json
{
  "stops": [...],
  "dropped": [{"name": "砍掉的站点", "reason": "原因"}]
}
```

### 效果

| 场景 | 修复前 | 修复后 | 关键变化 |
|------|--------|--------|---------|
| 目的地型 | B 78 | **S 92** | LLM 学会了"长隆玩一天1站就够" |
| 特种兵型 | F 42 | **A 85** | 时间可行从0→75 |

**教训**：在加架构之前，先试 prompt。LLM 其实能学会拒绝，只是你从没教过它。把"你有权拒绝"写在 prompt 最显眼的位置，它真的会用。

## 第五阶段：时间分配的空窗和倒流

### 现象

特种兵型10站路线，第8站 14:45 离开，第9站 17:00 到达 — 中间 2 小时 15 分钟空窗，什么都没安排。时间可行 0 分。

### 原因

`_llm_fix_times` 完全信任 LLM 返回的时间，没有任何验证。LLM 给出 2 小时空窗，就保留 2 小时空窗。

### 修复

新增 `_smooth_times()` 后处理函数，在 LLM 返回时间后做算法修正：

1. **检测空窗**：如果某段交通时间 >45min，按坐标重新估算（~3min/km + 5min缓冲），重新计算到达时间
2. **检测倒流**：如果后站到达时间 < 前站离开时间，修正为前站离开 + 交通时间
3. **检测溢出**：如果总时间超出 end_time，等比压缩所有停留时间（最多压到50%）
4. **最终兜底**：压缩后仍溢出则截断到 end_time

```python
def _smooth_times(steps, start_time, end_time):
    for i in range(1, len(steps)):
```

### 效果

特种兵型时间可行从 0 → 75，总分从 F 42 → A 85。

**教训**：LLM 的时间分配不可信。它没有物理时钟，不知道 15 分钟走不了 3 公里。LLM 出方案，算法做验证。

## 第六阶段：美食型的路线膨胀

### 现象

美食型场景被 cap 到 6 站，但 ensure 函数又把站数加回 9 站。9 站里有咖啡书屋、书店、书吧 — 跟美食毫无关系。

### 原因

pipeline 里有 4 个 ensure 函数依次执行：

```
_cap_route_stops (截到6站)
→ _ensure_food_in_route (补餐饮)
→ _ensure_poi_in_route (补景点) ← 问题在这
→ _ensure_min_food_in_route (补餐饮)
→ _ensure_food_scene_food_count (补餐饮子类)
```

`_ensure_poi_in_route` 不区分场景类型，美食型也往路线里补非餐饮 POI。ensure 保护条件是 `len(steps) >= 10`，但美食型 cap 是 6，ensure 可以加到 9。

### 修复

- 美食型场景跳过 `_ensure_poi_in_route`（美食路线以餐饮为主，不需要强制补景点）
- `_ensure_food_scene_food_count` 加站数上限：`len(steps) + len(extra) >= 8: break`

### 效果

美食型从 F 35 → B 72。5站全餐饮，无多余POI。

**教训**：ensure 函数是"好意办坏事"的典型。每个 ensure 单独看都合理，组合起来就膨胀。解决方案不是砍 ensure，而是让每个 ensure 知道自己的边界。

## 第七阶段：目的地型的地理阈值

### 现象

目的地型路线：长隆海洋王国 → 澳美居茶餐厅（12km）。评分 LLM 正确指出"12km 不应该出现在目的地型路线中"。

### 原因

`_geo_reroute` 的跨区跳跃阈值是 15km，12km 不触发重排。目的地型的补充景点应该离核心目的地更近。

### 修复

- 目的地型用更严格的 geo_reroute 阈值（10km vs 默认15km）
- 长隆 → 12km 外的茶餐厅会被识别为"跳跃"，触发最近邻重排

### 效果

目的地型从 B 78 → S 92/95（两轮测试都稳定在 S 级）。

## 其他修复（前序对话中完成的）

### 目的地坐标扩展（5 → 25）

`expert_router.py` 和 `destination_expert.py` 的 `_DEST_COORDS` 从 5 个扩展到 25 个。之前只有长隆、御温泉、圆明新园、梦幻水城，缺海泉湾、港珠澳大桥、东澳岛等。目的地不在列表里 → destination_expert 返回空 → 走通用 poi_expert → 选出跨区 POI。

### LLM 评分 + 规则平滑

从 3 次取中位数改回单次 LLM + 规则平滑。3 次 LLM 调用导致 API 限流（NotEnoughCvError）。改为：如果 LLM 分和规则分偏差 >30，用 60% LLM + 40% 规则混合。

### Intent Parser 位置提取

规则解析路径缺少位置提取。加了 `_LOCATION_KW_MAP` 覆盖 12 个区域关键词（横琴、金湾、斗门、唐家湾等），`_extract_location()` 做 rule-based fallback。

## 核心教训

1. **先怀疑数据采集，再怀疑算法**：评分 0 分不是路线烂，是评分表印错了。地理跳跃不是真跳跃，是评分 LLM 看不到坐标。

2. **LLM 不知道的事情会编**：不给坐标它就凭名字猜距离，猜错了还自信。给它事实，它就能判对。

3. **先试 prompt，再加架构**：教 LLM "你有权拒绝" 写了 6 行 prompt，效果堪比一个 feasibility gate 组件。别过度设计。

4. **LLM 的时间分配不可信**：它没有物理时钟，不知道 15 分钟走不了 3 公里。LLM 出方案，算法做验证。

5. **ensure 函数是好意办坏事**：每个单独看都合理，组合起来就膨胀。让每个 ensure 知道自己的边界。

6. **关键匹配要分层**："凉茶"包含"茶"但不包含"茶餐厅"。先精确后模糊。

7. **先跑通再优化**：3 次取中位数的想法很美，API 限流就废了。单次 + 规则平滑够用。
