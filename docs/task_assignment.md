# CityFlow 任务分工方案

## 背景
当前系统正确率 63%（19/30），有 11 个失败场景需要分析和改进。

---

## 同学 A：技术架构分析（会编程）

### 任务 1：架构问题分析（2天）
**输入**：`docs/failed_cases_analysis.md`

**输出**：`docs/architecture_issues.md`

**内容**：
```markdown
## 架构层面问题分析

### 1. FeasibilityAgent 漏检问题
- 现象：4个场景应拒绝但未拒绝
- 根因：检测逻辑不够严格
- 方案：修改 FeasibilityAgent，核心需求缺失时强制拒绝

### 2. Solver 地理连续性缺陷
- 现象：美食场景跨区 27km
- 根因：缺乏场景感知约束
- 方案：美食场景限制 10km 搜索半径

### 3. 营业时间过滤不完善
- 现象：深夜场景返回已关门 POI
- 根因：business_hours 字段不准确
- 方案：深夜场景强制 24h 营业过滤
```

### 任务 2：构建自动化测试集（1天）
**输出**：`eval_data/regression_tests.json`

**要求**：
- 30 个边界场景（营业时间/预算/地理/意图冲突）
- 每个场景标注：预期结果（通过/拒绝）
- 提供测试脚本 `scripts/run_regression.py`

### 任务 3：代码实现（3天）
**修改文件**：
- `backend/agents/__init__.py` - 强化 FeasibilityAgent
- `backend/services/solver.py` - 地理约束
- `backend/services/filters.py` - 营业时间过滤

## 同学 B：产品体验测试（不会编程）

### 任务 1：人工审核失败案例（1天）
**输入**：`docs/failed_cases_analysis.md`

**输出**：`docs/human_review.md`

**要求**：
对每个失败场景，从真实用户角度判断：
1. 这个需求合理吗？
2. 系统给出的路线能用吗？
3. LLM 的评分公平吗？

**示例**：
```markdown
## 案例：朋友轰趴 (overall=6)

用户角度：5个朋友人均150，想玩+吃
系统路线：餐饮→狼人杀→餐饮，总价249元

判断：
- 需求合理性：✅ 非常合理，常见场景
- 路线可用性：⚠️ 超预算但可接受（狼人杀95元偏贵）
- LLM评分公平性：❌ 偏低！14.7km被扣分，但规则说<25km正常

建议：这个评分 6 分偏低，应该给 7 分
```

### 任务 2：设计真实用户场景（2天）
**输出**：`eval_data/real_user_scenarios.json`

**要求**：
- 20 个场景
- 基于真实生活经验（旅行/约会/亲子/团建）
- 不考系统，考"人觉得是否合理"

**示例**：
```json
{
  "name": "周末闺蜜逛街",
  "input": "两个女生周末下午逛街，想买衣服和喝奶茶，预算200",
  "why_realistic": "常见闺蜜活动，需求明确",
  "expected_outcome": "购物+餐饮路线"
}
```

### 任务 3：竞品对比测试（1天）
**输出**：`docs/competitor_analysis.md`

**要求**：
用同样的需求测试：
- 美团
- 高德地图
- 小红书

对比它们的推荐和我们的推荐，记录差异。
