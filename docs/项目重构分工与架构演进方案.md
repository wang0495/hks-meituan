## 三、三人分工逻辑

### 写在前面：为什么要这样分

- **西瓜妹看不懂代码** — 这是优势，不是劣势。因为 LLM 评测最大的坑就是"写代码的人自己给自己打分"，默认觉得自己写得好。西瓜妹代表真实用户视角，她来定标准才靠谱。
- **范林刚入门** — "测试法庭"是最佳切入点：不需要写 Agent，但每天看 Agent 的产出、评 Agent 的好坏、追着乐天派改 Bug，比看教程学得快 10 倍。
- **乐天派** — 核心架构你守住，接口你来定，剩下的放手让他们长。

### 3.1 西瓜妹 → 路线质量的定义者

**一句话定位：** 你说什么是好路线，AI 就学什么是好路线。

**为什么是你？**
LLM 评测的核心瓶颈从来不是代码，而是 **"评分标准是否对得起真实用户"**。一个不懂代码但较真的人，比十个工程师更适合定义"好"的标准。

**你的产出（不需要写代码）：**

| 你要做的事 | 产出的文件 | 文件里写什么 |
|-----------|-----------|------------|
| 定义评分标准 | `docs/specs/评分标准.md` | 好路线有几条标准（比如：路线合不合理、POI 好不好吃、时间顺不顺），每条标准多少分，什么算及格 |
| 定义场景类型 | `docs/specs/场景分类.md` | 用户有几种需求（一个人安静走 / 情侣约会 / 带娃出去玩 / 朋友聚餐 / 特种兵打卡），每种需求好路线的标准不一样 |
| 生成测试用例 | `docs/specs/测试用例.md` | 20~30 条"用户说的话"，比如"周末想一个人安静走走"，你觉得 AI 应该推荐什么路线才算对 |
| 优化评分 Prompt | `docs/prompts/评分指令v2.md` | 你写中文指令，告诉 AI "按什么标准打分、每个标准怎么算分" |
| 用户画像扩展 | `docs/specs/用户画像V3.md` | 现在的 20 组画像够不够？要不要加"大学生穷游""一人食探店"？ |

**第一周任务书（具体到每一步）：**

| # | 做什么 | 怎么做 | 花多久 | 需要谁帮 |
|---|--------|--------|--------|---------|
| 1 | 了解现在系统怎么评分 | 让乐天派打开电脑，给你看一遍当前测试报告的截图（30 分钟），你记笔记 | 半天 | 乐天派 |
| 2 | 写场景分类 v1 | 用 Word/记事本，回答：用户有几种类型？每种最看重什么？（例如：带娃的看重安全、约会的看重氛围） | 1 天 | — |
| 3 | 改 3 条评分 Prompt | 打开 `prompts/scoring/` 目录下的文件，把里面你觉得"AI 听不懂"的话改成大白话 | 1 天 | 乐天派告诉你文件在哪 |
| 4 | 跑一次评分，看结果 | 范林帮你运行测试，你把评分结果截图贴到文档里，写上"这个我同意 / 这个我不服" | 半天 | 范林 |
| 5 | 提交第一版标准 | 把上面所有东西放到 `docs/specs/` 文件夹里，用微信告诉乐天派"写完了" | 半天 | — |

**6 周成长路线（不写代码）：**
```
第 1 周: 学会"AI 是怎么打分"的 → 改 3 条 Prompt
第 2 周: 学会"怎么防止 AI 乱打分" → 设计评分标准
```

### 3.3 乐天派 → 架构掌舵人

**你守住的三条线：**
1. 所有模块间的接口契约（`backend/contracts/*.py`）— 西瓜妹和范林的产出靠这些接口喂给系统
2. 核心引擎 (M2/M3/M4/M5) — Agent 编排、Harness、Context 管理，这些不能放
3. 小模型替代架构演进 — 三步走路线图你来把控节奏

**你的原则：**
- 西瓜妹说"我觉得这条路线不好" → 你要问她"你觉得哪里不好？标准是什么？"而不是直接改代码
- 范林说"这个 Bug 我发现了" → 你要赞他，然后当天修掉
- 接口定好就别改 → 改接口要全组知会，跑全量回归

## 四、Context 管理规范 (乐天派制定, 全员遵守)

### 4.1 压缩时机

```
Agent 编排流中, 在以下节点主动触发上下文压缩:

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ rule_guard  │ ──→ │expert_router│ ──→ │  专家并行    │
│ 结束后压缩   │     │ 结束后压缩   │     │ 各自结束后压缩  │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                                               ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ live_itin.  │ ←── │ synthesizer  │ ←── │ review/rework │
│ (不用压缩)   │     │ 结束后压缩   │     │ 每次review前压缩│
└─────────────┘     └─────────────┘     └─────────────┘
```

## 五、三周极限方案：只做能落地的事

### 战略原则

只有 3 周，砍掉一切"锦上添花"的东西：

| 做 | 不做 |
|---|------|
| 西瓜妹能立刻上手的事 | 等她学完 6 周再干活 |
| 范林能立刻上手的事 | 搞复杂的架构重构 |
| 替换已有的规则引擎（0 成本） | 引入新模型（Qwen API 还要申请） |
| 砍 LLM 调用次数（最快降本） | 换模型（最慢、最危险） |
| 先跑通、再优化 | DSA / Agent 缓存 |

### 目标

- **成本**: LLM 调用次数砍 50%（从 8 次降到 3-4 次）
- **延迟**: 从 40s+ 降到 20s 以内
- **协作**: 三人都有核心产出，不是打杂
- **基准**: 通过率不降

### 基准锁定（第一天就做）

```
① 跑一次全量回归 → 记录 baseline.json
   { pass_rate, avg_score, per_scene_scores, avg_latency }
② baseline.json 提交到 git, tag 为 baseline_v1
③ 后续每次改动后:
   范林跑回归 → 对比 baseline → 通过率降 > 3% 立刻回滚
```

### 具体技术方案

#### 第 1 周（上手周）— 乐天派：先止血

目标：把现成的规则引擎捡起来，立刻砍掉 3 个不必要的 LLM 调用。

```
当前 LLM 调用链路 (8 次):
  rule_guard(1次) → expert_router(1次)
  → POI专家(1次) + 美食专家(1次) + 交通(1次) + 天气(1次) + ...
  → review(1次) → synthesizer(1次)

第 1 周目标 (5 次):
  rule_guard(1次) → expert_router(1次)
  → POI筛选(规则引擎) + 美食推荐(规则引擎 + 小模型可选)
  → review(规则引擎) → synthesizer(1次)
```

| 替换环节 | 当前 LLM 调用 | 替换为 | 代码位置 | 风险 |
|---------|:-:|--------|---------|:---:|
| POI 候选筛选 | 1 次 deepseek | `_smart_poi_selection()` 规则引擎 | `agents_v3/experts/poi_expert.py` | 极低：规则已经存在，只是没启用 |
| 文案生成 | 1 次 deepseek | `generate_narrative()` 模板引擎 | `services/narrator.py` | 极低：模板已存在，降级模式 |
| 核心 POI 确保 | 1 次 deepseek | 关键词正则 | `agents_v3/nodes/rule_guard.py` | 低：有现成兜底函数 |
| Review | 1 次 deepseek | 确定性规则检查 | `agents_v3/nodes/review.py` | 中：改用规则后检查力度降级，但 review 本身不是必须的 |

**总计砍掉次数：8 → 4**，成本降低 **50%**，延迟预计从 40s → 20s。

#### 第 2-3 周（优化周）— 乐天派：专家路由规则化

| 替换环节 | 风险 | 方案 |
|---------|:---:|------|
| expert_router (场景分类 + 专家激活) | 中 | 用 `_rule_precheck()` 覆盖 80% 的常见输入（"珠海美食一日游"→ 直接匹配），剩下的 20% 模糊输入再调 LLM |
| 美食专家 | 低 | 用已有的 `_food_intent_hint` + POI 数据库的 `category` 字段筛选，加上评分排序即可 |
| 交通专家 | 零 | 保持现状（0 LLM 调用，纯 Haversine 计算）|
| 天气/住宿/预算黑客 | 零 | 保持现状（规则引擎或 0 LLM）|

**最终调用链 (3 次)：**
```
rule_guard(1次) → expert_router(规则/LLM混合 → 80%情况走规则)
→ 全部专家走规则引擎 (0 LLM)
→ review(规则检查, 0 LLM)
→ synthesizer(1次 deepseek)  ← 保留，这是最关键的
```

| 指标 | 当前 | 三周后 |
|------|:---:|:-----:|

| 端到端延迟 | 40s+ | **< 20s** |
| 西瓜妹参与度 | 边缘 | 负责评分标准 + 测试用例 + 评审 |
| 范林参与度 | 边缘 | 负责前端可视化 + 回归测试 + Bug 追踪 |
| 架构文档 | 无 | 接口契约定义完毕 |
| 基准测试 | 无基线 | baseline_v1 锁定 |
