# Harness工程：CityFlow测试体系建设

## 什么是Harness工程？

简单说：搭一套自动化测试框架，像工厂质检一样，每个改动都能快速验证好坏。

类比：
- 你改了下刹车系统 → 马上跑一遍碰撞测试
- 你改了下引擎 → 马上测油耗和加速
- 不能等用户撞了才发现问题

---

## 为什么要精细化管理？

### 现状问题
| 问题 | 后果 |
|------|------|
| 只有30个测试场景 | 改完代码不敢保证没破坏其他功能 |
| 场景靠人工想 | 容易遗漏边界情况（如凌晨3点） |
| 测试靠手工跑 | 每次改完测1小时，效率低 |
| 不知道为啥失败 | 是代码bug？还是LLM评分不准？ |

### 精细化管理目标
1. **分层测试**：单元测试（模块）→ 集成测试（Agent协作）→ 端到端（完整流程）
2. **维度覆盖**：时间/预算/地理/人群/天气/意图冲突
3. **渐进扩容**：10→30→50→100→1000，每阶段都有意义
4. **自动验证**：改完代码10分钟出结果

---

## 如何构建正确评估系统？

### 三层评估

```
Layer 1: 单元测试（同学A做）
├── 测试单个函数是否正常
├── 例：营业时间过滤函数
│   输入：凌晨2点，POI营业时间09:00-22:00
│   预期：返回False（不开门）
│   不需要LLM，代码直接断言
└── 运行时间：<1秒

Layer 2: Agent协作测试（同学A做）
├── 测试多个Agent配合是否正常
├── 例：IntentAgent说可行→FeasibilityAgent也说可行→才能继续
│   输入："想蹦迪"
│   预期：FeasibilityAgent检测无酒吧→返回infeasible
│   不需要LLM评分，看事件流
└── 运行时间：5-10秒

Layer 3: 端到端体验测试（同学B做）
├── 测试完整流程，LLM评分
├── 例：从输入到路线生成到评分
│   输入："情侣约会，预算300"
│   预期：overall>=6.5才算通过
│   需要LLM评分（LongCat，免费）
└── 运行时间：30-60秒/场景
```

### 为什么分三层？

| 层级 | 发现问题 | 修复成本 |
|------|----------|----------|
| Layer 1 | 函数算错了 | 5分钟 |
| Layer 2 | Agent配合错了 | 30分钟 |
| Layer 3 | 整体体验差 | 可能几小时 |

**原则**：尽量在Layer 1发现问题，别等到Layer 3才发现。

## 如何构建测试集？

### 维度矩阵（必须覆盖）

```
时间维度：
├── 凌晨（00:00-06:00）→ 测试24h营业过滤
├── 早上（06:00-12:00）→ 测试日出场景
├── 下午（12:00-18:00）→ 测试常规场景
└── 晚上（18:00-24:00）→ 测试夜景/夜市

预算维度：
├── 零预算（0元）
├── 低预算（<100元）
├── 中预算（100-500元）
└── 高预算（>500元，不限）

人群维度：
├── 独居
├── 情侣
├── 亲子（带小孩）
├── 朋友（3-5人）
└── 老人

意图冲突维度（最难）：
├── 预算vs需求冲突（50元住五星）
├── 时间vs需求冲突（3小时玩遍全城）
├── 天气vs需求冲突（下雨户外烧烤）
└── 人群vs需求冲突（带6岁娃蹦迪）
```

### 测试集构建方法（避免拍脑袋）

**方法1：边界值法**（同学B负责想场景）
- 不只是"预算100"，要"预算99"和"预算101"
- 不只是"凌晨2点"，要"23:59"和"00:01"
- 为什么要这样？因为代码经常在边界出错

**方法2：等价类划分**（同学A负责分类）
- 把相似场景归为一类，每类抽一个代表
- 例：所有"看日出"场景本质一样，测一个就够了
- 例：但"看日出+一个人"和"看日出+带娃"不同，都要测

**方法3：故障注入**（同学A负责构造）
- 故意构造系统应该拒绝的场景
- 例："预算50住五星"→应该拒绝
- 例："凌晨3点去博物馆"→应该拒绝
- 如果系统没拒绝，就是bug

## 如何避免测试集混入训练集？

### 问题背景
- 系统里用了LLM（DeepSeek/LongCat）
- LLM在训练时可能已经见过类似场景
- 如果测试场景和训练数据重复，分数会虚高

### 解决方法

**方法1：场景变形**（同学B做）
- 不要直接抄网上的旅游攻略
- 把常见场景改得"奇怪"一点
- 例：正常是"情侣路看日落"，改成"🌅💑💰0求方案"（emoji形式）

**方法2：时空错位**（同学B做）
- 用冷门城市或虚构组合
- 例：不要测"北京故宫"，测"珠海横琴看极光"（珠海没极光，看系统怎么反应）
- 例："凌晨4点喝早茶"（广东人知道早茶是早上，但系统应该检测时间不合理）

**方法3：对抗生成**（同学A做）
- 用代码自动生成"奇葩"场景
- 随机组合：时间+预算+人群+需求
- 例："零预算+带狗+蹦迪+凌晨3点"
- 这些组合现实中不存在，LLM训练时肯定没见过

## 渐进式测试体系

### 阶段1：10个场景（Day 1-2）
**目的**：快速验证框架能跑通

**场景选择**：最基础的5个 + 最离谱的5个
- 基础：情侣约会、亲子出游、朋友聚餐、老人散步、一个人看海
- 离谱：50元住五星、凌晨3点蹦迪、零预算浪漫一天、3小时玩遍全城、下雨天户外烧烤

**通过标准**：8/10通过，框架OK，继续下一阶段

---

### 阶段2：30个场景（Day 3-5）
**目的**：覆盖主要维度

**场景来源**：
- 已有11个失败案例
- 同学B写的20个真实场景

**测试方式**：
- 每次改完代码，跑这30个
- 10分钟内出结果
- 记录正确率变化趋势

**通过标准**：正确率>=70%，且没有"应该拒绝却给路线"的严重bug

---

### 阶段3：50个场景（Week 2）
**目的**：边界情况全覆盖

**新增场景**：
- 时间边界：23:59, 00:00, 00:01, 05:59, 06:00
- 预算边界：0, 1, 99, 100, 101, 499, 500, 501
- 地理边界：同一区、跨区、跨市

**自动化程度**：
- 全部自动化跑
- 生成可视化报告（正确率趋势图）

**通过标准**：正确率>=75%，错误案例有明确分类（是bug还是数据问题）

---

### 阶段4：100个场景（Week 3-4）
**目的**：压力测试，发现隐藏bug

**生成方式**：
- 代码自动生成（随机组合维度）
- 人工审核，确保不过于离谱

**重点测试**：
- Agent协作是否稳定
- 极端预算/时间下是否崩溃
- 并发性能（同时跑多个场景）

**通过标准**：正确率>=80%，无崩溃/超时

---

### 阶段5：1000个场景（Month 2+）
**目的**：长期回归测试

**运行方式**：
- 每晚自动跑一遍
- 早上出报告
- 发现 regression（新代码破坏旧功能）立即告警

**抽样策略**：
- 不必等1000个全跑完
- 随机抽100个，10分钟出结果
- 全量1000个周末跑
