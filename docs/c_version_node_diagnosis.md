# C版本节点级架构图 + 瓶颈根因分析

> 核心结论：**问题不在算法，在 prompt 和传给 Agent 的信息。**
> geo_continuity 低是因为 coordinator prompt 没把地理约束说清楚。
> 候选池一样是因为评分公式没有场景分化。
> POI离群是因为 POI Agent prompt 没告诉 LLM 要选同区域的。

## ② POI Agent 架构图（问题核心）

```
candidates(~120个, 来自rule_guard)
    │
    │  ⚠️ 注意: 这120个对每个场景已经不一样了
    │     (filter_candidates按预算/时间/硬约束做了过滤)
    │     但! 排序后会高度重叠（见下方）
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│  POI Agent                                                          │
│                                                                     │
│  Step 1: scored_candidates 评分排序                                  │
│  ┌─────────────────────────────────────────────────────────────────┐│
│  │ 对每个候选POI计算总分:                                           ││
│  │                                                                 ││
│  │ ① rating * 0.3          ← 最高 4.5×0.3 = 1.35                  ││
│  │ ② is_tourist * 1.5      ← 0 或 1.5                             ││
│  │ ③ llm_quality * 0.2     ← 1.0~2.0                              ││
│  │ ④ _LANDMARK_NAMES +3.0  ← 🔴 这个是问题！                      ││
│  │ ⑤ tag_similarity * 3.0  ← 0~3.0                                ││
│  │ ⑥ tags数 * 0.2          ← 0~1.0                                ││
│  │ ⑦ 用户提及 * 5.0        ← 只在用户说了具体景点时才加             ││
│  │ ⑧ 亲子/情侣适合度 +1.5  ← 🔴 这个权重太低了！                  ││
│  │                                                                 ││
│  │ 实际评分对比 (情侣场景):                                         ││
│  │ ┌──────────────────────────┬────────┬────────┬─────────┐       ││
│  │ │ POI                      │ 地标分 │ 总分   │ 排名    │       ││
│  │ ├──────────────────────────┼────────┼────────┼─────────┤       ││
│  │ │ 情侣路海滨(r=4.5,地标)   │ +3.0   │ 7.55   │ ①      │       ││
│  │ │ 梅溪牌坊(r=4.3,地标)     │ +3.0   │ 7.39   │ ②      │       ││
│  │ │ 珠海渔女(r=3.9,地标)     │ +3.0   │ 6.47   │ ③      │       ││
│  │ │ 野狸岛(r=4.2,地标)       │ +3.0   │ 7.25   │ ④      │       ││
│  │ │ 某浪漫咖啡厅(r=4.8,非)   │ +0     │ 4.94   │ 25+    │       ││
│  │ │ 某艺术画廊(r=4.5,非)     │ +0     │ 4.85   │ 30+    │       ││
│  │ └──────────────────────────┴────────┴────────┴─────────┘       ││
│  │                                                                 ││
│  │ 🔴 ④ > ①+②+③ 的总和!                                          ││
│  │    一个rating=3.0的地标(总分~5.7)                               ││
│  │    压过rating=5.0的非地标(总分~5.25)                            ││
│  │                                                                 ││
│  │ 🔴 ⑧ 亲子/情侣适合度只有+1.5，比地标+3.0低一半                 ││
│  │    场景相关性 < 地标知名度                                       ││
│  │    → POI Agent的LLM收到的候选已经是偏向地标的                    ││
│  └─────────────────────────────────────────────────────────────────┘│
│    │                                                                │
│    ▼ top_candidates = scored_candidates[:30]                        │
│    │                                                                │
│    │ 🔴 这30个对每个场景都差不多:                                    │
│    │   情侣场景 top30: 情侣路① 梅溪牌坊② 野狸岛③ 渔女④ ...        │
│    │   美食场景 top30: 情侣路① 梅溪牌坊② 野狸岛③ 渔女④ ...        │
│    │   特种兵场景 top30: 情侣路① 梅溪牌坊② 野狸岛③ 渔女④ ...      │
│    │                                                                │
│    │ 唯一的区别: tag_similarity * 3.0 可能微调几个位置               │
│    │ 但3.0对地标+3.0来说只是杯水车薪                                 │
│    │                                                                │
│    ▼                                                                │
│  Step 2: 构建poi_summaries → 传给LLM                                │
│    (30个候选的name/category/rating/price/tags/lat/lng)               │
│    │                                                                │
│    ▼                                                                │
│  Step 3: LLM选择景点                                                │
│    system prompt = _build_poi_prompt(intent) ← 有场景分化!          │
│    user prompt = 候选POI(30个) + 用户需求                           │
│    │                                                                │
│    │ 🟢 prompt本身没问题,分化做得挺好:                               │
│    │   亲子 → "优先选有儿童游乐设施、海洋馆、动物园"                 │
│    │   情侣 → "优先选浪漫氛围、海滨、观景台"                         │
│    │   特种兵 → "选最知名景点，覆盖不同区域"                         │
│    │                                                                │
│    │ 🔴 但LLM看到的30个候选高度同质！                                │
│    │   就算prompt说"选浪漫咖啡馆"，候选里排前面的全是地标            │
│    │   LLM只能在地标里挑，跳不出这个圈                               │
│    │                                                                │
│    ▼                                                                │
│  Step 4: _geo_cluster_filter 地理聚类后处理                          │
│    ┌─────────────────────────────────────────────────────────────────┐│
│    │ 🔴🔴 这个后处理才是最大的问题                                   ││
│    │                                                                 ││
│    │ 本意: LLM可能选了分散的POI，把离群的替换成簇中心附近的           ││
│    │ 实际: 替换了LLM精心选择的跨区域POI                              ││
│    │                                                                 ││
│    │ 特种兵场景:                                                     ││
│    │   LLM选了: 渔女(香洲) + 海洋王国(横琴) + 圆明新园(吉大)        ││
│    │   这其实是对的！特种兵就是要跨区域打卡                           ││
│    │                                                                 ││
│    │   但geo_cluster_filter算簇中心:                                 ││
│    │     center ≈ (22.24, 113.54)                                    ││
│    │     海洋王国(22.11) 距中心>15km → 判定为离群!                   ││
│    │                                                                 ││
│    │   然后用簇中心10km内的候选替换:                                  ││
│    │     替换评分也用地标+3.0 → 替换进来的又是市区地标                ││
│    │                                                                 ││
│    │ 🔴 问题: POI Agent的prompt根本没说"选同区域的POI"              ││
│    │   如果prompt说了"注意地理紧凑，所有景点应在10km半径内"           ││
│    │   LLM自己就不会选横琴+香洲混合的                                ││
│    │   或者如果说了"特种兵需要跨区域覆盖"                             ││
│    │   那就不应该做geo_cluster替换                                    ││
│    │                                                                 ││
│    │ 🔴 geo_cluster_filter 是在补 prompt 的课                       ││
│    │   但补的方式太粗暴: 直接替换掉LLM的选择                         ││
│    │   正确做法: 让prompt自己约束地理范围，不用后处理暴力替换         ││
│    │   或者: geo_cluster只在亲子/闲逛等需要紧凑的场景才生效          ││
│    │          特种兵场景不应该做geo聚类（本来就要跨区域）             ││
│    └─────────────────────────────────────────────────────────────────┘│
│    │                                                                │
│    ▼                                                                │
│  输出: proposals (5-8个景点)                                         │
└─────────────────────────────────────────────────────────────────────┘
```

## ⑨ group_debate 架构图

```
7个Agent的proposals合并 (~15个提案)
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│  group_debate                                                       │
│                                                                     │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │ Step 1: 规则层冲突检测 (纯规则, 不调LLM)                       │ │
│  │                                                                │ │
│  │ ① 主题矛盾: 亲子+夜生活 → remove                              │ │
│  │ ② 预算超支: 总价>1.5x预算 → suggest_cheaper                   │ │
│  │ ③ 地理矛盾: 餐厅距景点簇太远 → remove_food                     │ │
│  │                                                                │ │
│  │ 🟢 这一步是合理的，规则冲突确实需要硬性处理                    │ │
│  └────────────────────────────────────────────────────────────────┘ │
│    │                                                                │
│    ▼ 执行规则决议 (remove的直接删除, suggest_cheaper的降confidence) │
│    │                                                                │
│    ▼                                                                │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │ Step 2: LLM结构化反驳 (1轮)                                    │ │
│  │                                                                │ │
│  │ 输入:                                                          │ │
│  │   - 各Agent提案 (含坐标、评分、推荐理由)                       │ │
│  │   - 规则层已检测到的冲突                                       │ │
│  │   - 用户需求/群体/预算                                         │ │
│  │                                                                │ │
│  │ LLM只能检查4种冲突:                                            │ │
│  │   1. geo_far: 景点间距>15km                                    │ │
│  │   2. type_repeat: 3+个同类                                     │ │
│  │   3. time_conflict: 时间冲突                                   │ │
│  │   4. scene_mismatch: 场景不匹配                                │ │
│  │                                                                │ │
│  │ 输出: debate_results (每个标记remove/swap/keep)                │ │
│  │                                                                │ │
│  │ 🟡 问题: 这个LLM能看到所有Agent的提案+坐标                    │ │
│  │   它能发现"渔女和海洋王国差了20km"这种问题                    │ │
│  │   但它只能标记，不能建议具体的替代方案                         │ │
│  │                                                                │ │
│  │ 🟡 swap_hint 太模糊:                                           │ │
│  │   LLM说"建议替换为同区域更合适的POI"                           │ │
│  │   但coordinator读_swap_hint后只降confidence                    │ │
│  │   不会真的去找替代POI                                          │ │
│  │                                                                │ │
│  │ 🟢 结构化设计是对的（1轮反驳，不做自由对话）                   │ │
│  │   问题在coordinator没接住swap_hint                             │ │
│  └────────────────────────────────────────────────────────────────┘ │
│    │                                                                │
│    ▼                                                                │
│  输出: cleaned proposals, negotiation_msgs, conflicts               │
│                                                                     │
│  ⚠️ group_debate本身没有大问题                                      │
│  真正的问题: 它检测到geo_far后标记swap                              │
│  但coordinator不处理swap（只处理remove）                            │
│  → swap标记的POI还在proposals里，只是confidence被降到0.2           │
│  → coordinator的LLM编排仍然看到这些低分POI                         │
└─────────────────────────────────────────────────────────────────────┘
```

## ⑩ coordinator LLM编排 — prompt问题分析

```
用户核心质疑:
  "给agent传递的地理位置信息约束说清楚，我不信他还来回跳"

当前coordinator._llm_assemble_route 的prompt:

┌─────────────────────────────────────────────────────────────────────┐

│                                                                     │
│  你是旅行路线编排专家。你需要把Agent精选的景点、餐厅、住宿          │
│  组合成一条完整的一日游路线。                                       │
│                                                                     │
│  你的任务（按优先级）:                                               │
│  1. 【地理连贯】同区域景点连走，避免折返。                         │

│  2. 【时间节奏】按情绪曲线设计...                                   │
│  3. 【餐饮就近】餐厅必须插在距它最近的景点旁边                      │
│  ...                                                                │
│                                                                     │
│  ═══════════════════════════════════════════════════════════════════ │
│  🔴 问题1: "同区域景点连走，避免折返" 太模糊                      │
│                                                                     │
│  LLM不知道什么是"同区域"。珠海有:                                  │

│    - 拱北 (lat≈22.22, 口岸/湾仔)                                   │

│                                                                     │
│  LLM看到:                                                           │
│    渔女→情侣路: 1.2km                                               │
│    渔女→野狸岛: 1.5km                                               │
│    ...                                                              │
│  LLM看不到:                                                         │
│    梅溪牌坊→野狸岛: 11km  ← 关键！被截掉了                        │
│                                                                     │
│  LLM不知道梅溪牌坊和野狸岛隔了11km                                 │
│  → 可能排成: 梅溪牌坊→野狸岛 (以为很近)                            │

│                                                                     │

│  如果prompt说:                                                      │
│    "珠海主要区域由南到北: 横琴(22.11)→拱北(22.22)→                │

│  ═══════════════════════════════════════════════════════════════════ │
│                                                                     │

│                                                                     │
│  景点Agent精选（5个）:                                              │
│  [                                                                  │
│    {"name":"珠海渔女","lat":22.265,"lng":113.583,...},              │
│    {"name":"梅溪牌坊公园","lat":22.31,"lng":113.482,...},           │
│    ...                                                              │
│  ]                                                                  │
│                                                                     │
│  景点间距离:                                                        │
│  [                                                                  │
│    {"from":"珠海渔女","to":"野狸岛","km":1.5},                      │
│    {"from":"珠海渔女","to":"情侣路","km":1.2},                      │
│    ... (被截断到25条)                                               │
│    🔴 梅溪牌坊↔野狸岛的11km距离被截掉了                            │
│  ]                                                                  │
│                                                                     │
│  交通Agent建议顺序: ["珠海渔女","野狸岛","梅溪牌坊",...]            │

│     编排Agent盲目参考 → 一起错                                      │
│                                                                     │
│  ═══════════════════════════════════════════════════════════════════ │
│  结论: 不是LLM能力不够，是信息没给够                                │
│                                                                     │
│  如果prompt加上:                                                    │
│    1. 珠海区域划分 + 坐标范围                                       │
│    2. 完整距离矩阵（不截断）                                       │
│    3. "路线必须单向，从lat小到大或从lat大到小"                      │
│                                                                     │
│  LLM完全能排出无折返路线                                            │
│  不需要2-opt后处理来补救                                            │
└─────────────────────────────────────────────────────────────────────┘
```

## 根因汇总：不是算法问题，是信息传递问题

```
                    ┌─────────────────────────────────────┐
                    │      信息传递链断裂点                 │
                    └─────────────────────────────────────┘

rule_guard ─────→ POI Agent ─────→ coordinator
   │                  │                   │
   │  candidates      │  top30给LLM       │  距离矩阵给LLM
   │  (~120个)        │  (同质化!)        │  (被截断!)
   │                  │                   │
   │                  │  prompt有分化      │  prompt说"避免折返"
   │                  │  但候选不分!       │  但没给区域划分!
   │                  │                   │
   │                  ▼                   ▼
   │            LLM只能在地标里挑    LLM不知道哪些点远
   │            → 5个场景选类似的    → 排出回头路
   │                                → geo=6.4
   │
   │  _ensure_key_pois
   │  默认5个地标
   │  但这步没病
   │  病在POI Agent
   │  的评分公式
   │
   ▼
┌─────────────────────────────────────────────────────────┐
│ 三个修复方向（都是prompt/信息层面，不涉及算法）:          │
│                                                         │
│ 1. POI Agent评分: 地标加分 +3.0 → +1.0                  │
│    tag_similarity * 3.0 → * 5.0                         │
│    亲子/情侣适合度 +1.5 → +3.0                          │
│    → 让场景相关性 > 地标知名度                           │
│    → 不同场景的top30自然会不同                           │
│                                                         │
│ 2. POI Agent prompt: 加上地理约束                       │
│    "选的景点应集中在同一区域(10km内)                     │
│     或明确覆盖不同区域（特种兵场景）"                    │
│    → LLM自己不会选横琴+香洲混合（除非特种兵）           │
│    → 不需要geo_cluster_filter暴力替换                   │
│                                                         │
│ 3. coordinator prompt: 加上区域锚点                     │
│    "珠海区域: 横琴(lat~22.11)→拱北(22.22)→              │
│     吉大(22.27)→香洲(22.30)→唐家湾(22.36)               │
│     路线应沿一个方向走"                                 │
│    + 距离矩阵不截断                                     │
│    → LLM能排出单向无折返路线                            │
│    → 不需要2-opt后处理                                  │
└─────────────────────────────────────────────────────────┘
```
