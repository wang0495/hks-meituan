# CityFlow A/B/C三版本对比测试报告

> 测试时间: 2026-05-12
> 评测方式: LLM自动评分 (LongCat-Flash-Lite), 0-10分制, 及格线6.5

---

## 1. 三版本架构概述

| 维度 | A版本 (3层联邦) | B版本 (Validator) | C版本 (分布式Agent) |
|------|----------------|-------------------|---------------------|
| 架构模式 | 中央调度+竞标市场+对抗校验 | LangGraph + Validator链 | 7自治Agent + LangGraph |
| 路线求解 | solver.solve_route() 5阶段 | solver.solve_route() 5阶段 | Agent协商涌现, coordinator组装 |
| POI选择 | 竞标Agent投标 → 评标 | 规则过滤 + 排序 | DeepSeek LLM决策 |
| 冲突处理 | 对抗校验层 + 微协商 | Validator规则链 | 涌现式校验 + Agent协商 |
| 智能来源 | 规则引擎 + LLM辅助 | 纯规则引擎 | DeepSeek V4 全LLM决策 |
| 节点数 | ~12 | ~8 | 11 (rule_guard + 7 agents + emergence + negotiation + coordinator) |

### C版本核心设计
```
用户 → rule_guard(意图+硬约束)
     → [7 Agent并行](全部DeepSeek LLM决策)
        ├─ POI Agent: LLM从候选中选景点(含坐标聚类)
        ├─ Food Agent: LLM选餐厅
        ├─ Hotel Agent: LLM判断需求+选酒店
        ├─ Traffic Agent: LLM规划交通顺序
        ├─ Weather Agent: LLM评估天气影响
        ├─ Local Expert Agent: LLM推荐隐藏景点
        └─ Insurance Agent: LLM分析行程风险
     → emergence_check(矛盾检测)
     → negotiation(Agent协商)
     → coordinator(提案组装, 不调用solver)
     → live_itinerary(热力图+溯源)
```

## 3. C版本SWOT分析

### 优势 (Strengths)

1. **POI质量高(6.6)** — DeepSeek LLM从候选池中挑选的POI质量普遍不错，0个低分(1-3)
2. **架构灵活** — 新增/修改Agent无需改动整体结构，Agent间松耦合
3. **可解释性好** — 每个Agent有独立的reasoning，可追溯决策过程
4. **情侣/浪漫场景表现好** — 穷游情侣、情侣热恋均获7分

### 劣势 (Weaknesses)

1. **地理连续性差(5.0)** — 最大短板。没有solver的5阶段优化(贪心+2-opt+呼吸空间+高潮+组装)，纯LLM选点+最近邻排序无法保证路线流畅
2. **意图匹配低(5.3)** — 24%场景intent≤3，特别是：
   - 特殊时段(摄影夜景 intent=2)
   - 矛盾需求(intent=2) — LLM无法同时满足冲突需求
   - 亲子场景(intent=2) — Agent不理解"带6岁孩子"的特殊需求
3. **横琴区域数据空白** — 长隆周边无餐饮数据，导致亲子场景被迫穿插市中心餐厅

### 机会 (Opportunities)

1. **8个差0.5分的场景** — 宠物同行(6), 雨夜漫步(6), 退休夫妇(6), 美食暴走(6)等，小幅提升即可通过
2. **Agent协商未充分利用** — negotiation_agent当前只做简单冲突检测，可以做更深层的Agent间协调
3. **LLM prompt优化** — 当前prompt未针对不同场景类型调整，可按场景分型优化

### 威胁 (Threats)

1. **LLM不稳定性** — 同一场景多次运行分数波动±1分
2. **API延迟** — 每场景需调用DeepSeek 5-7次，平均40秒/场景
3. **成本** — 30场景约消耗DeepSeek API 150-200次调用

### 关键差异

2. **C版本在极端场景表现差** — 30场景测试暴露了C版本处理特殊需求的能力不足
3. **C版本延迟最高** — 全LLM决策导致每场景45秒(vs A的8秒)
4. **C版本POI质量最高** — 6.6分, 0个低分POI

## 5. 结论与建议

### 5.1 总体评价

C版本（分布式智能体网络）在**标准旅游规划场景**中表现不输甚至略优于A/B版本，证明了"Agent协商涌现路线"架构的可行性。但在**非标准/极端场景**中明显落后，主要受限于：

1. 无solver优化的地理连续性不足
2. LLM对复杂/矛盾意图的理解不稳定
3. 全LLM决策带来的延迟和成本

### 5.2 C版本优化方向

| 优先级 | 优化项 | 预期收益 |
|--------|--------|----------|
| P0 | coordinator接入solver路线优化 | geo_continuity 5.0→7.0+ |
| P1 | 按场景类型分化POI Agent prompt | intent_match 5.3→6.5+ |
| P2 | 横琴区域补充餐饮数据 | 亲子场景 5→7+ |
| P3 | negotiation_agent增强协商能力 | 矛盾需求场景 4→6+ |

### 5.3 架构选型建议

- **追求稳定性和速度** → A版本(3层联邦) + solver
- **追求可解释性和灵活性** → C版本(分布式Agent) + solver辅助
- **当前最佳实践** → A+C混合: 用Agent协商选POI, 用solver优化路线
