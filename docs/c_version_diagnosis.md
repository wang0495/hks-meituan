## 2. 问题汇总（按影响排序）

```
影响度排序:

🔴🔴 P0 ── coordinator 没接入 solver
│   影响: geo_continuity 5.0→7.0+ (预计总分 +0.7~1.0)
│   原因: 只有最近邻+简单2-opt，缺少时间窗/疲劳/情绪编排
│   修复: coordinator中调用 solver.solve_route(candidates, intent)
│
🔴 P1 ── POI Agent 选点地理分散
│   影响: 横琴+市中心来回跳, geo=3
│   原因: LLM选POI时虽然给了坐标，但仍经常跨区选
│   修复: geo_cluster_filter已加(🟢), 但20km阈值仍不够紧
│
🟡 P2 ── intent_match 低 (5.4/10, 24%≤3分)
│   影响: hard/extreme场景完全理解不了用户意图
│   原因: prompt是通用模板，不按场景类型分化
│   修复: 按group_type/pace/特殊需求分化prompt
│
🟡 P3 ── 横琴区域无餐饮数据
│   影响: 亲子/长隆场景插不进合适的餐厅
│   原因: data_service中横琴POI缺餐饮
│   修复: 补充横琴餐饮数据 or 长隆场景跳过餐饮插入
│
🟡 P4 ── Agent协商是假的
│   影响: 矛盾需求场景(4分)无法调和
│   原因: negotiation_agent只做一次评估，不迭代
│   修复: 改为多轮协商或接入A版本的对抗校验层
│
🟢 P5 ── 延迟高 (45s/场景)
│   影响: 用户体验差
│   原因: 7个Agent串行调DeepSeek (虽然graph并行，LLM API限流)
│   修复: Agent复用LLM连接 or 减少Agent数量
```

## 4. 修复优先级与预期收益

| 序号 | 问题 | 当前 | 修复后预期 | 优先级 | 改动量 |
|------|------|------|-----------|--------|--------|
| 5 | coordinator接入solver | geo=5.0 | geo=7.0+ | **🔴🔴 P0** | 5行代码 |
| 1 | intent prompt分化 | intent=5.4 | intent=6.5+ | 🟡 P1 | 中等 |
| 2 | POI评分加地理权重 | geo分散 | 更紧凑 | 🟡 P2 | 小 |
| 3 | 横琴餐饮数据 | 亲子=5 | 亲子=7+ | 🟡 P3 | 数据 |
| 4 | Traffic看POI结果 | 顺序不准 | 更准 | 🟢 P4 | 架构改动 |

**只做修复5（接入solver），预计30场景总分从5.3→6.0+，通过率从14%→30%+。**
